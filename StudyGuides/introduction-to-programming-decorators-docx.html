Big Picture Pages 1-2

A decorator is a function that takes another function and returns a replacement object, usually a wrapper function. The wrapper adds behavior before, after, or around the original function call without editing the original function body.

Core exam sentence: A decorator is executed when the decorated function is defined; the wrapper is executed later when the function name is called.

The PDF motivates decorators with debugging/logging: if many functions should print messages on entry and exit, editing every function is repetitive or impossible if you do not control the code. A decorator centralizes that behavior.

Source idea What it means in Python
Functions can be returned from functions. An outer function can create an inner function object and return it to the caller.
Functions are first-class objects. A function object can be stored in a variable, passed as an argument, returned, and attached to attributes.
The function name is a pointer/reference. The name is bound to a function object. Rebinding the name changes what the name points to.
A decorator creates a wrapper. The original function is passed into another function. The returned wrapper replaces the original name.

Functions as Objects Page 1

The PDF starts with two prerequisite ideas: nested functions and function assignment.

def f(x):
    def sq(z):
        return z * z
    return sq(x)

g = f
print(g(3))

Output:

9

Behind the scenes

  1. Executing def f(x): ... creates a function object and binds the name f to it.
  2. g = f does not copy the function. It binds g to the same function object.
  3. g(3) calls that function object with x = 3.
  4. Each call to f creates a new local name sq bound to a new inner function object.
  5. sq(x) calls the inner function immediately and returns 9.
Trap from page 1: The name of a function is not sacred. If you write sum = 0, you bind the name sum to an integer in that scope, hiding the built-in sum function. Then sum([1, 2, 3]) fails because sum now refers to an int, not a function. 
sum = 0
print(sum([1, 2, 3]))

Result: TypeError: 'int' object is not callable.

Basic Wrapper Decorator Pages 2-3

The first decorator example wraps a no-argument function:

def my_decorator(func):
    def wrapper():
        print("Something is happening before the function is called.")
        func()
        print("Something is happening after the function is called.")
    return wrapper

def say_whee():
    print("Whee!")

say_whee = my_decorator(say_whee)
say_whee()

Output:

Something is happening before the function is called.
Whee!
Something is happening after the function is called.

What each name points to

Step Binding
After def say_whee say_whee points to the original function object.
During my_decorator(say_whee) func points to the original say_whee function.
After return my_decorator returns wrapper.
After assignment The name say_whee now points to wrapper, not directly to the original function.

General wrapper with arguments

For functions like add_numbers(x, y), the wrapper must accept arbitrary positional and keyword arguments:

def log_function_call(func):
    def wrapper(*args, **kwargs):
        print(f"Calling function {func.__name__}")
        result = func(*args, **kwargs)
        print(f"Finished calling function {func.__name__}")
        return result
    return wrapper

@log_function_call
def add_numbers(x, y):
    return x + y

result = add_numbers(2, 3)
print(result)

Output:

Calling function add_numbers
Finished calling function add_numbers
5

*args collects positional arguments like 2, 3. **kwargs collects keyword arguments like x=2, y=3. The wrapper forwards them with func(*args, **kwargs).

@ Syntax and Name Binding Pages 3-6

The PDF states the general syntax:

@decorate
def func(x):
    ...

This is shorthand for:

def func(x):
    ...

func = decorate(func)
Source correction: Page 3 says add_numbers = log_function_call(add_numbers(2,3)). The intended equivalent is add_numbers = log_function_call(add_numbers). You pass the function object to the decorator, not the result of calling the function.

Why the corrected version matters

# Correct:
add_numbers = log_function_call(add_numbers)

# Wrong for this decorator:
add_numbers = log_function_call(add_numbers(2, 3))

In the wrong version, add_numbers(2, 3) runs first and returns 5. Then Python tries to decorate 5 as if it were a function. The wrapper later tries func.__name__ or func(*args, **kwargs), which fails because func is an integer.

Decorator placement rule

Decorators must appear on their own line immediately before the function definition:

@my_decorator
def my_function(x, y):
    return x + y

Behind the Scenes Pages 7-8

Decoration happens at definition time, not call time.

def my_decorator(func):
    def wrapper():
        print("Something is happening before the function is called.")
        func()
        print("Something is happening after the function is called.")
    return wrapper

@my_decorator
def say_hello():
    print("Hello!")

# At this point, say_hello is already decorated
say_hello()

Exact order of events

  1. Python executes def my_decorator(func): ... and binds the name my_decorator to a function object.
  2. Python sees @my_decorator above say_hello.
  3. Python creates the original say_hello function object.
  4. Immediately, Python calls my_decorator(original_say_hello).
  5. Inside my_decorator, Python creates the inner wrapper function object.
  6. wrapper closes over func, meaning it keeps a reference to the original say_hello function.
  7. my_decorator returns wrapper.
  8. The name say_hello is bound to wrapper.
  9. Later, say_hello() calls wrapper().
  10. wrapper() prints before, calls the original function through func(), then prints after.
source code: @my_decorator def say_hello(): ...
definition time: original say_hello object -> my_decorator(original) -> wrapper
name binding: say_hello -> wrapper
call time: say_hello() -> wrapper() -> func() -> original say_hello()

Metadata and functools.wraps Pages 4-6

The PDF shows a practical problem: the wrapper hides the original function's metadata.

def my_decorator(func):
    def wrapper(*args, **kwargs):
        """ wrapper"""
        print("Before the function is called.")
        result = func(*args, **kwargs)
        print("After the function is called.")
        return result
    return wrapper

@my_decorator
def my_function(x, y):
    """ my func"""
    return x + y

z = my_function(2, 3)
print(z, my_function.__name__)
print(z, my_function.__doc__)

Output without wraps:

Before the function is called.
After the function is called.
5 wrapper
5  wrapper

The name my_function now points to the wrapper, so my_function.__name__ and my_function.__doc__ are the wrapper's metadata.

The fix

import functools

def my_decorator(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        """ wrapper"""
        print("Before the function is called.")
        result = func(*args, **kwargs)
        print("After the function is called.")
        return result
    return wrapper

@my_decorator
def my_function(x, y):
    """ my func"""
    return x + y

z = my_function(2, 3)
print(z, my_function.__name__)
print(z, my_function.__doc__)

Output with wraps:

Before the function is called.
After the function is called.
5 my_function
5  my func

functools.wraps(func) is itself a decorator applied to wrapper. It copies metadata such as __name__, __doc__, annotations, and module information from func to wrapper. It also sets __wrapped__ so tools can find the original function.

Applying the same decorator to another function

Pages 6 shows the same decorator used on mult:

@my_decorator

z = mult(2, 3)
print(z, mult.__name__)

Before the function is called.
After the function is called.

Multiple Decorators Pages 8-9

When multiple decorators appear above a function, the decorator closest to the function is applied first. At call time, the outermost wrapper runs first.

def decorator1(func):
    def wrapper(*args, **kwargs):
        print("Decorator 1")
        return func(*args, **kwargs)
    return wrapper

def decorator2(func):
    def wrapper(*args, **kwargs):
        print("Decorator 2")
        return func(*args, **kwargs)
    return wrapper

def decorator3(func):
    def wrapper(*args, **kwargs):
        print("Decorator 3")
        return func(*args, **kwargs)
    return wrapper

@decorator3
@decorator2
@decorator1
def my_function():
    print("Hello, World!")

my_function()

Equivalent binding:

my_function = decorator3(decorator2(decorator1(my_function)))

Output:

Decorator 3
Decorator 2
Decorator 1
Hello, World!
Phase Order
Application at definition time decorator1, then decorator2, then decorator3.
Execution at call time decorator3.wrapper, then decorator2.wrapper, then decorator1.wrapper, then original function.
Professor trap: Students often reverse the output because they memorize "closest first." Closest first is application order. Printed output follows wrapper execution order, which starts with the outermost wrapper.

Function Objects, Memory, and References Pages 10-11

The PDF emphasizes that the decorated function is normally a different object from the original function.

  1. Python creates the original function object.
  2. The decorator receives a reference to that object.
  3. The decorator returns a new object, usually wrapper.
  4. The original function name is rebound to the returned object.
  5. The wrapper may keep a reference to the original through a closure.

Keeping access to the original

def my_decorator(func):
    def wrapper(*args, **kwargs):
        print("Decorator adds this before the function call")
        result = func(*args, **kwargs)
        print("Decorator adds this after the function call")
        return result

    wrapper.original = func
    return wrapper

@my_decorator
def original_function(x, y):
    print(f"Original function called with arguments: {x}, {y}")
    return x + y

print("Calling decorated function:")
original_function(5, 10)

print("\nCalling undecorated function:")
original_function.original(5, 10)

Output:

Calling decorated function:
Decorator adds this before the function call
Original function called with arguments: 5, 10
Decorator adds this after the function call

Calling undecorated function:
Original function called with arguments: 5, 10

wrapper.original = func attaches the original function object as an attribute of the wrapper function object. Because functions are objects, they can have attributes.

Memory/reference behavior: The original function is not destroyed as long as some live reference still points to it. The wrapper's closure variable func and the explicit wrapper.original attribute both keep references to the original function.

Decorator Factory: countcalls Pages 12-13

The PDF asks for a decorator that keeps track of how many times each decorated function has been called, using a dictionary passed to the decorator.

calldict = {}

@countcalls(calldict)
def add(x, y):
    return x + y

@countcalls(calldict)
def mult(x, y):
    return x * y

z = add(2, 3)
print(calldict)
z = add(2, 3)
print(calldict)
z = mult(2, 3)
print(calldict)

Required output:

{'add': 1}
{'add': 2}
{'add': 2, 'mult': 1}

Solution from the PDF

def countcalls(calldict):
    def decorator(func):
        def wrapper(*args, **kwargs):
            calldict[func.__name__] = calldict.get(func.__name__, 0) + 1
            return func(*args, **kwargs)
        return wrapper
    return decorator

Why three nested functions?

Function Job
countcalls(calldict) Receives the dictionary and returns a real decorator. This is a decorator factory.
decorator(func) Receives the function being decorated and returns a wrapper.
wrapper(*args, **kwargs) Runs when the decorated function is called. It increments the count and calls the original function.

Definition-time order

@countcalls(calldict)
def add(x, y):
    return x + y

is equivalent to:

def add(x, y):
    return x + y

add = countcalls(calldict)(add)
  1. countcalls(calldict) runs first and returns decorator.
  2. The original add function object is created.
  3. decorator(add) runs and returns wrapper.
  4. The name add is rebound to wrapper.
  5. Only when add(2, 3) is called does the dictionary count change.
Trap: The dictionary is shared because both decorated functions close over the same calldict object. The decorator does not create a separate dictionary per function unless you pass or create separate dictionaries.

Timing Code: time.time and timeit.timeit Pages 13-16

The PDF introduces timing before building a timer decorator.

Tool Use it for How it works
time.time() General timing and longer-running tasks. Returns current seconds since the Unix Epoch. Measure duration by subtracting start from end.
timeit.timeit() More accurate timing of short snippets or benchmarks. Runs code many times in a controlled environment and returns total time.

General timing pattern

import time

start = time.time()
# code to time
end = time.time()
print(f"Duration: {end - start} seconds")
Source correction: Page 15 prints {end start}. The correct expression is {end - start}.

Timing a sleeping function

import time

def my_function():
    time.sleep(2)
    return "Hello, world!"

start_time = time.time()
result = my_function()
end_time = time.time()
duration = end_time - start_time

print(f"Result: {result}")
print(f"Duration: {duration} seconds")

The sample PDF run gives about 2.0110862255096436 seconds. The exact number changes by machine and run.

Using timeit with a callable

import timeit

def my_function():
    for i in range(1000000):
        pass

time_taken = timeit.timeit(my_function, number=100)
print("Execution time:", f"{time_taken:.4f} seconds")

number=100 means the function is called 100 times and the returned duration is the total time for all 100 runs.

Do not shadow sum in real code

Page 16 uses:

def my_function():
    sum = 0
    for i in range(1000000):
        sum += i
    return sum

This works locally inside the function, but it shadows the built-in sum. For style and safety, a name like total is better.

Timer Decorator for Sorting Algorithms Pages 17-19

The PDF asks for a timer decorator that decorates a sort function, sorts a list, and returns the runtime. The answer returns both runtime and the sorted array.

import time
import functools

def timer(func):
    @functools.wraps(func)
    def wrapper_timer(*args, **kwargs):
        start_time = time.time()
        value = func(*args, **kwargs)
        end_time = time.time()
        runtime = end_time - start_time
        return runtime, value
    return wrapper_timer

When a sort is decorated with @timer, calling the sort no longer returns just the sorted list. It returns a tuple:

(runtime, sorted_value)

Source-specific sort functions

The PDF says "bubble sort, insertion sort, quicksort and merge sort" on page 17, but the code implements bubble_sort, selection_sort, merge_sort, and quick_sort. There is no insertion sort implementation in the shown answer.

Bubble sort

Mutates the input list by repeatedly swapping adjacent out-of-order elements.

@timer
def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr

Selection sort

Mutates the input list by selecting the smallest remaining element and swapping it into position.

@timer
def selection_sort(arr):
    for i in range(len(arr)):
        min_idx = i
        for j in range(i + 1, len(arr)):
            if arr[min_idx] > arr[j]:
                min_idx = j
        arr[i], arr[min_idx] = arr[min_idx], arr[i]
    return arr

Recursive decorated functions are tricky

In the PDF, merge_sort and quick_sort are decorated. That means recursive calls to merge_sort or quick_sort call the wrapper, not the undecorated function.

@timer
def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[0]
    less = [x for x in arr[1:] if x <= pivot]
    greater = [x for x in arr[1:] if x > pivot]
    _, sorted_less = quick_sort(less)
    _, sorted_greater = quick_sort(greater)
    return sorted_less + [pivot] + sorted_greater

The unpacking is necessary because each recursive call returns (runtime, sorted_list). If the code wrote sorted_less = quick_sort(less), then sorted_less would be a tuple, not a list.

Trap: Decorating a recursive function changes what its own name means inside its body. After decoration, the global name quick_sort points to wrapper_timer. Recursive calls through that name run the timer wrapper too.

Testing setup

data_size = 10000
test_data = [random.randint(1, 10000) for _ in range(data_size)]

times = {}
times["Bubble Sort"], _ = bubble_sort(test_data.copy())
times["Selection Sort"], _ = selection_sort(test_data.copy())
times["Merge Sort"], _ = merge_sort(test_data.copy())
times["Quick Sort"], _ = quick_sort(test_data.copy())

test_data.copy() matters because some algorithms mutate the list. Without copying, later algorithms might receive already-sorted data and the comparison would be unfair.

The PDF plots two bar charts with matplotlib: one normal scale and one logarithmic y-axis using plt.yscale("log"). A log scale helps compare runtimes when slow algorithms are much slower than fast algorithms.

Flow of Control Example Pages 19-20

The final PDF example manually decorates a function instead of using @ syntax:

def hello_decorator(func):
    def inner1():
        print("Hello, this is before function execution")
        func()
        print("This is after function execution")
    return inner1

def function_to_be_used():
    print("This is inside the function !!")

function_to_be_used = hello_decorator(function_to_be_used)
function_to_be_used()

Output:

Hello, this is before function execution
This is inside the function !!
This is after function execution

Step-by-step control path

  1. hello_decorator is defined.
  2. function_to_be_used is defined.
  3. hello_decorator(function_to_be_used) is called.
  4. Inside hello_decorator, func points to the original function_to_be_used.
  5. inner1 is created and returned.
  6. The name function_to_be_used is rebound to inner1.
  7. function_to_be_used() now calls inner1().
  8. inner1 prints before, calls func(), then prints after.

Common Traps from This PDF

1. Passing a function call instead of a function object

decorate(func) passes the function object. decorate(func()) calls the function first and passes its return value.

2. Forgetting to return the wrapper

If the decorator does not return wrapper, the decorated function name becomes None. Calling it gives TypeError: 'NoneType' object is not callable.

3. Forgetting to return the original result from the wrapper

If wrapper calls func() but does not return its result, the decorated function returns None.

4. Wrapper accepts no arguments

A wrapper written as def wrapper(): fails for add(2, 3). Use *args, **kwargs for general decorators.

5. Confusing definition time and call time

The decorator function runs during function definition. The wrapper body runs when the decorated function is called.

6. Multiple decorator order

Applied bottom-up. Executed top-down.

7. Metadata hidden without functools.wraps

Without @functools.wraps(func), __name__ and __doc__ usually show wrapper metadata.

8. Decorated recursive functions return decorated results

If a timer returns (runtime, value), recursive calls also return (runtime, value) unless you deliberately call an undecorated reference.

9. Mutating test data

Sorting functions that mutate the input should receive copies when comparing runtimes.

Professor-Style Questions with Answers

Question 1: What prints? 
def deco(func):
    print("A")
    def wrap():
        print("B")
        func()
        print("C")
    print("D")
    return wrap

@deco
def f():
    print("E")

print("F")
f()

Output:

A
D
F
B
E
C

Reasoning: When f is defined, deco(f) runs immediately, so A and D print before F. The wrapper body does not run until f(). At call time, f points to wrap, so B, original E, then C print.

Question 2: What is wrong with this decorator? 
def deco(func):
    def wrapper(*args, **kwargs):
        print("before")
        func(*args, **kwargs)
        print("after")
    return wrapper

@deco
def add(x, y):
    return x + y

print(add(2, 3))

Output:

before
after
None

Reasoning: add now calls wrapper. The wrapper calls the original add, but it does not return that result. A Python function with no explicit return returns None.

Fix:

result = func(*args, **kwargs)
return result

Question 11: Why does this recursive quicksort unpack? 
@timer
def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[0]
    less = [x for x in arr[1:] if x <= pivot]
    greater = [x for x in arr[1:] if x > pivot]
    _, sorted_less = quick_sort(less)
    _, sorted_greater = quick_sort(greater)
    return sorted_less + [pivot] + sorted_greater

Because quick_sort is decorated, every call to quick_sort returns (runtime, value), not just the sorted list. The underscore receives the runtime that the algorithm does not need for the recursive combination step.

Question 12: What happens if the wrapper has no *args?

Error:

Reasoning: After decoration, add points to wrapper. So add(2, 3) calls wrapper(2, 3), but wrapper accepts no parameters.

Question 13: Closure memory behavior 
def deco(func):
    def wrapper():
        return func()
    return wrapper

@deco
def f():
    return "ok"

print(f())

Output: ok.

Reasoning: Even though the name f was rebound to wrapper, the wrapper has a closure cell containing a reference to the original function object. That reference keeps the original function alive and callable through func().

Question 15: Likely exam extension: write a decorator that prints arguments

Write show_args so this works:

@show_args
def add(x, y=0):
    return x + y

One good answer:

import functools

def show_args(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print("args:", args)
        print("kwargs:", kwargs)
        return func(*args, **kwargs)
    return wrapper

Expected output:

args: (3,)
kwargs: {'y': 4}
7

This is naturally testable from the PDF because it uses the same *args, **kwargs forwarding pattern as the logging and timing decorators.
