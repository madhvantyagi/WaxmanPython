Big Picture

The PDF is not teaching OOP from scratch. It is mostly teaching how familiar C++ class features map into Python's object model.

C++ idea

  • private and public enforce access.
  • Multiple constructors can have different signatures.
  • Operator overloads are named with operators, like operator+.
  • Memory management can require constructors, copy constructors, assignment operators, and destructors.

Python idea

  • Privacy is convention-first: _name means internal; __name triggers name mangling.
  • One __init__; simulate alternatives with defaults, *args, or @classmethod.
  • Operator overloads use special methods like __add__ and __getitem__.
  • Objects live on the heap and are cleaned up by reference counting plus garbage collection in CPython.
Exam framing: the professor can ask, "What does this print?" but the real skill is knowing which special method is called, which object owns the data, and whether a name lookup hits the instance, class, property descriptor, or fallback right-hand operator.

Rat Class Translation pp. 1-4

The PDF starts with a C++ rational-number class and then translates it to Python. The same object idea exists in both: store numerator and denominator, validate denominator, support arithmetic, and print cleanly.

Clean Python version of the slide idea

class Rat:
    def __init__(self, n=0, d=1):
        self.n = n
        self.d = d

    @property
    def n(self):
        return self._n

    @n.setter
    def n(self, value):
        self._n = value

    @property
    def d(self):
        return self._d

    @d.setter
    def d(self, value):
        if value == 0:
            raise ValueError("Denominator cannot be zero")
        self._d = value

    def __add__(self, other):
        if isinstance(other, Rat):
            return Rat(self.n * other.d + self.d * other.n,
                       self.d * other.d)
        elif isinstance(other, int):
            return self + Rat(other)
        return NotImplemented

    def __radd__(self, other):
        return self + other

    def __sub__(self, other):
        if isinstance(other, Rat):
            return Rat(self.n * other.d - self.d * other.n,
                       self.d * other.d)
        elif isinstance(other, int):
            return self - Rat(other)
        return NotImplemented

    def __rsub__(self, other):
        return Rat(other) - self

    def __mul__(self, other):
        if isinstance(other, Rat):
            return Rat(self.n * other.n, self.d * other.d)
        elif isinstance(other, int):
            return self * Rat(other)
        return NotImplemented

    def __rmul__(self, other):
        return self * other

    def __truediv__(self, other):
        if isinstance(other, Rat):
            return Rat(self.n * other.d, self.d * other.n)
        elif isinstance(other, int):
            return self / Rat(other)
        return NotImplemented

    def __rtruediv__(self, other):
        return Rat(other) / self

    def __str__(self):
        return f"{self.n}/{self.d}"
Source-specific correction: the slide code on page 3 uses def __rsub__(self, other): return -self + other, but the shown class does not define __neg__. If you run that exact version, 5 - Rat(1, 2) raises a TypeError. The safer implementation is return Rat(other) - self.

Operator Overload Dispatch

Expression Method Python tries Why
r1 + r2 r1.__add__(r2) Left operand is a Rat, so its __add__ owns the first attempt.
r1 + 3 r1.__add__(3) The slide handles int by converting 3 into Rat(3).
3 + r1 int.__add__(3, r1), then r1.__radd__(3) If the left operand cannot handle the right operand, Python asks the right operand's reflected method.
r1 / r2 r1.__truediv__(r2) In Python 3, / maps to __truediv__, not __div__.
print(r1) str(r1), which calls r1.__str__() This replaces the C++ << stream overload in this example.

Behind the scenes for r = Rat(3, 4)

  1. Python looks up the class object named Rat.
  2. Calling a class normally invokes Rat.__new__(Rat) first. Since the slide does not define __new__, Python uses object.__new__ to allocate a blank object.
  3. Python then calls Rat.__init__(blank_object, 3, 4).
  4. Inside __init__, self.n = n does not simply put n into the instance dictionary because n is a property. It calls the n setter, which stores self._n = value.
  5. self.d = d calls the denominator setter. The setter rejects zero and stores self._d = value if valid.
  6. The variable r receives a reference to the new object. The object holds references to two integer objects through its attributes.

What arithmetic returns

The slide does not reduce fractions. That matters for output questions.

r1 = Rat(3, 4)
r2 = Rat(2, 5)
print(r1 + r2)  # 23/20
print(r1 - r2)  # 7/20
print(r1 * r2)  # 6/20, not 3/10
print(r1 / r2)  # 15/8

Constructors, self, and cls p. 5

The PDF compares C++ multiple constructors with Python's single __init__. Python only keeps one method with a given name in a class body. If you write two __init__ methods, the later definition replaces the earlier one.

Three Python ways to imitate multiple constructors

Technique Slide example Exam issue
Default arguments def __init__(self, n=0, d=1) Rat(), Rat(5), and Rat(3, 4) all work through one method.
Class methods @classmethod def from_string(cls, string) cls points to the class used in the call, so subclasses can construct subclass instances.
Variable argument lists SA.__init__(self, *args) on pages 17-18 Code branches on len(args). Invalid argument counts must be rejected manually.

self is explicit

The slide says self is like C++ this, but must be explicitly written. More precisely:

  • self is not a keyword. You could name it this, x, or anything else, but self is the standard convention.
  • When you call r.__str__(), Python passes r as the first argument automatically.
  • When you define the method, you must include that first parameter yourself.
class Demo:
    def show(self):
        print(self)

d = Demo()
d.show()       # Python turns this into Demo.show(d)
Demo.show(d)   # Same call, written explicitly

cls in class methods

class Rat:
    def __init__(self, n=0, d=1):
        self.n = n
        self.d = d

    @classmethod
    def from_string(cls, string):
        n, d = string.split("/")
        return cls(int(n), int(d))

class SpecialRat(Rat):
    pass

x = SpecialRat.from_string("3/4")
print(type(x).__name__)  # SpecialRat
Why this matters: using return Rat(n, d) inside the class method would always build a Rat. Using return cls(n, d) preserves subclass behavior, which is exactly the distinction called out on page 5.

Data Hiding and Properties pp. 6-8

The C++ version uses private. Python does not enforce the same kind of privacy. Instead, the PDF gives three levels of access discipline.

Access levels in Python

Pattern Meaning Enforced? Example
name Public attribute. No restriction. obj.name
_name Internal-use convention. No. Other code can still access it. obj._hidden_data
__name Name-mangled to reduce accidental collisions. Partly. It is harder to access accidentally, not truly private. obj._ClassName__name

Properties are descriptors

The slide describes properties as getter, setter, and deleter methods. Behind the scenes, a property object is stored on the class. It implements descriptor behavior, so attribute access can call methods instead of directly reading or writing instance data.

class Person:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if value < 0:
            raise ValueError("Age cannot be negative")
        self._age = value

p = Person(20)
print(p.age)   # calls Person.age.__get__(p, Person), then getter
p.age = 21     # calls Person.age.__set__(p, 21), then setter

Why the slide's Rat uses self.d = d, not self._d = d

Inside __init__, assigning self.d = d deliberately goes through the property setter. That means the constructor gets the same denominator validation as later assignments.

r = Rat(1, 2)
r.d = 0        # raises ValueError
Rat(1, 0)      # also raises ValueError because __init__ uses self.d = d
Trap: properties do not hide the backing attribute absolutely. A determined caller can still write r._d = 0 and break the rational-number invariant. The point is controlled normal access, not absolute security.

Instance Variables, Class Variables, and Static Methods pp. 8-11

Instance variables

Instance variables belong to one object. In the Rat class, every object has its own numerator and denominator.

r1 = Rat(1, 2)
r2 = Rat(3, 4)
r1.n = 9
print(r1)  # 9/2
print(r2)  # 3/4

Class variables

Class variables live on the class object and are shared by normal lookup across instances.

class Rat:
    count = 0

    def __init__(self, n=0, d=1):
        self.n = n
        self.d = d
        Rat.count += 1

r1 = Rat(1, 2)
r2 = Rat(3, 4)
print(Rat.count)  # 2

Attribute lookup order

  1. For ordinary attributes, Python first checks the instance dictionary: obj.__dict__.
  2. If not found, Python checks the class and parent classes.
  3. Properties and other data descriptors on the class can intercept lookup before instance data.
class C:
    x = 10

a = C()
b = C()
print(a.x, b.x, C.x)  # 10 10 10
a.x = 99              # creates an instance attribute on a
print(a.x, b.x, C.x)  # 99 10 10
C.x = 7
print(a.x, b.x, C.x)  # 99 7 7

Static methods

A static method belongs to the class namespace but does not receive self or cls automatically. The slide uses it for class-related utility behavior.

class Rat:
    count = 0

    def __init__(self, n=0, d=1):
        self.n = n
        self.d = d
        Rat.count += 1

    @staticmethod
    def get_total_rats_created():
        return Rat.count

print(Rat.get_total_rats_created())
Static vs class method trap: page 10 returns Rat.count. That is fine for exactly Rat, but it is less flexible for subclasses. If the method needs to respect subclasses, use @classmethod and cls.count.

Property Deleters pp. 11-13

The PDF emphasizes that a deleter does not change the definition of the class. It only defines what happens when a specific property is deleted from a specific instance.

class MyClass:
    def __init__(self, value):
        self._my_property = value

    @property
    def my_property(self):
        return self._my_property

    @my_property.setter
    def my_property(self, value):
        if value < 0:
            raise ValueError("Value cannot be negative")
        self._my_property = value

    @my_property.deleter
    def my_property(self):
        print("Property is being deleted")
        del self._my_property

What happens on deletion

obj = MyClass(5)
print(obj.my_property)
del obj.my_property
print(obj.my_property)

Output and result:

5
Property is being deleted
AttributeError on the final print
  1. print(obj.my_property) calls the getter and returns 5.
  2. del obj.my_property calls the deleter.
  3. The deleter deletes the backing attribute _my_property from obj.__dict__.
  4. The class still has the my_property property object.
  5. The final getter call tries to return self._my_property, but that instance attribute is gone.

Reasons the PDF gives for deleters

  • Cleanup external resources.
  • Prevent memory/resource leaks in resource-heavy classes.
  • Maintain state consistency.
  • Enforce business logic around deletion.
  • Log or audit deletion.
  • Validate whether a property should be deletable.
Exam warning: deleting attributes can make instances of the same class have different shapes. The PDF specifically warns that setting a value to None can be clearer than deleting the attribute outright.

Safe Array Translation pp. 14-18

The second source example translates a C++ safe-array class into Python. The C++ class manually handles bounds, heap allocation, copying, assignment, destruction, indexing, and printing. The Python class keeps the important behavior but uses a list for storage.

Python version from the slide idea

class SA:
    def __init__(self, *args):
        if len(args) == 0:
            self.low = 0
            self.high = -1
            self.data = []
        elif len(args) == 1:
            self.low = 0
            self.high = args[0] - 1
            self.data = [0] * args[0]
        elif len(args) == 2:
            l, h = args
            if h - l + 1 <= 0:
                raise ValueError("constructor error in bounds definition")
            self.low = l
            self.high = h
            self.data = [0] * (h - l + 1)
        else:
            raise ValueError("Invalid number of arguments")

    def __copy__(self):
        copy_obj = SA(self.high - self.low + 1)
        copy_obj.data = self.data.copy()
        return copy_obj

    def __getitem__(self, index):
        if index < self.low or index > self.high:
            raise IndexError(f"index {index} out of range")
        return self.data[index - self.low]

    def __setitem__(self, index, value):
        if index < self.low or index > self.high:
            raise IndexError(f"index {index} out of range")
        self.data[index - self.low] = value

    def __str__(self):
        return str(self.data)

What the constructor branches mean

Call Bounds Stored list Valid user indexes
SA() low = 0, high = -1 [] None
SA(10) low = 0, high = 9 10 zeroes 0 through 9
SA(10, 20) low = 10, high = 20 11 zeroes 10 through 20

Index translation

The external index is not always the internal list index. For SA(10, 20), external index 15 maps to internal list index 15 - 10 = 5.

sa = SA(10, 20)
sa[15] = 100
print(sa)       # [0, 0, 0, 0, 0, 100, 0, 0, 0, 0, 0]
print(sa[15])   # 100
print(sa.data[5])  # 100

C++ memory management vs Python memory management

C++ SA Python SA
int* p points to manually allocated array memory. self.data references a Python list object.
Constructor uses new int[size]. Constructor uses [0] * size to build a list.
Destructor uses delete [] p. No explicit destructor is needed for this list-backed version.
Copy constructor and assignment operator must make deep copies to avoid shared raw pointers. Assignment shares references. Use copy.copy, copy.deepcopy, or a custom copy method when a separate list is required.
operator[] returns int&, so assignment like x[15] = 100 works. __setitem__ handles assignment, and __getitem__ handles reading.
Source-specific copy trap: the slide's __copy__ creates SA(self.high - self.low + 1). For SA(10, 20), that copy has bounds 0..10, not 10..20. A bounds-preserving copy should create SA(self.low, self.high).

Python Memory and Reference Mechanics

These mechanics are not always spelled out in the slides, but they are directly testable from the Rat and SA examples.

Variables are references

a = SA(3)
b = a
b[1] = 99
print(a)  # [0, 99, 0]

a and b refer to the same SA object. No copy was made by assignment.

Object attributes reference other objects

An SA instance has attributes like low, high, and data. The data attribute references a list object. Mutating that list changes the same list object that the SA object points to.

sa = SA(3)
lst = sa.data
lst[0] = 7
print(sa)  # [7, 0, 0]

Integer immutability vs list mutability

Integers are immutable. When self._n = value stores an integer, later arithmetic creates new integer objects rather than changing the old integer. Lists are mutable. self.data[index] = value changes the existing list object.

Garbage collection and destructors

For the slide's Python SA, there is no explicit destructor because the list is a Python object. In CPython, objects are usually cleaned up when their reference count reaches zero, with a cycle detector for reference cycles. You should not rely on exact destructor timing for ordinary course-level Python code.

Important memory model consequences

  • x = y binds another name to the same object.
  • SA.__copy__ needs to copy the internal list if independent storage is required.
  • self.data.copy() is enough for a list of integers, but not enough for nested mutable objects.
  • del obj.prop removes a reference from the instance if the deleter deletes a backing attribute.
  • When no references remain to an object, the object becomes eligible for cleanup.

Professor-Style Output Questions

These questions are written from the exact PDF concepts and include the kind of step-by-step reasoning needed on a hard Python exam.

Q1. What prints? Rat arithmetic without simplification. 
r1 = Rat(3, 4)
r2 = Rat(2, 5)
print(r1 + r2)
print(r1 * r2)

Output:

23/20
6/20

Reasoning: r1 + r2 calls __add__. Numerator is 3 * 5 + 4 * 2 = 23. Denominator is 4 * 5 = 20. Multiplication returns Rat(3 * 2, 4 * 5), which is 6/20. The slide class does not reduce fractions.

Q2. What prints? Left-hand int uses reflected method. 
r = Rat(1, 2)
print(3 + r)

Output:

7/2

Reasoning: Python first tries int.__add__(3, r). Since an int cannot add a Rat, Python tries r.__radd__(3). The slide's __radd__ returns self + other, so r.__add__(3) converts 3 into Rat(3, 1). Result is 1 * 1 + 2 * 3 = 7 over 2 * 1 = 2.

Q3. What happens? Exact slide version of __rsub__. 
# Using the page 3 version:
# def __rsub__(self, other):
#     return -self + other

r = Rat(1, 2)
print(3 - r)

Result: it raises TypeError in the exact shown class.

Reasoning: 3 - r eventually calls r.__rsub__(3). That tries -self. The shown class does not define __neg__, so unary minus is unsupported for Rat. A corrected reflected subtraction is return Rat(other) - self.

Q4. What prints? Constructor defaults. 
print(Rat())
print(Rat(5))
print(Rat(5, 6))

Output:

0/1
5/1
5/6

Reasoning: Python does not use three separate constructors. One __init__(self, n=0, d=1) handles all three calls through default arguments.

Q5. What happens? Property validation in constructor. 
r = Rat(4, 0)
print(r)

Result: ValueError: Denominator cannot be zero.

Reasoning: __init__ does self.d = d, and d is a property. Assignment calls the setter. The setter rejects zero before print can run.

Q6. What prints? Bypassing the property. 
r = Rat(1, 2)
r._d = 0
print(r)

Output:

1/0

Reasoning: The property setter only runs on r.d = value. Directly assigning r._d bypasses validation. This demonstrates the PDF's point: Python does not enforce absolute privacy.

Q7. What prints? Class variable and instance shadowing. 
class C:
    count = 0

a = C()
b = C()
a.count = 5
C.count = 2
print(a.count, b.count, C.count)

Output:

5 2 2

Reasoning: a.count = 5 creates an instance variable on a. It does not change C.count. Then C.count = 2 changes the class variable. a.count finds the instance value first; b.count falls back to the class value.

Q8. What prints? Static method reading class variable. 
class Rat:
    count = 0

    def __init__(self):
        Rat.count += 1

    @staticmethod
    def get_total_rats_created():
        return Rat.count

print(Rat.get_total_rats_created())
a = Rat()
b = Rat()
print(a.get_total_rats_created())

Output:

0
2

Reasoning: A static method receives neither self nor cls. It can still be called from the class or an instance. Here it explicitly reads Rat.count.

Q9. What prints? Safe Array index translation. 
sa = SA(10, 20)
sa[15] = 100
print(sa[10])
print(sa[15])
print(sa[20])
print(sa)

Output:

0
100
0
[0, 0, 0, 0, 0, 100, 0, 0, 0, 0, 0]

Reasoning: Valid external indexes are 10..20. 15 maps to internal list position 5. Printing calls __str__, which returns str(self.data).

Q10. What happens? Out-of-range Safe Array access. 
sa = SA(10, 20)
print(sa[9])

Result: IndexError: index 9 out of range.

Reasoning: __getitem__ checks index < self.low or index > self.high. Since 9 < 10, it raises before reading the list.

Q11. What prints? Assignment shares the object. 
a = SA(3)
b = a
b[0] = 8
print(a)
print(b)

Output:

[8, 0, 0]
[8, 0, 0]

Reasoning: Python assignment does not call a copy constructor. a and b are two names for the same object. b[0] = 8 mutates that object.

Q12. What prints? The slide's __copy__ bounds issue. 
import copy

sa = SA(10, 20)
sa[15] = 100
cp = copy.copy(sa)
print(cp.low, cp.high)
print(cp)

Output with the slide's Python version:

0 10
[0, 0, 0, 0, 0, 100, 0, 0, 0, 0, 0]

Reasoning: __copy__ calls SA(self.high - self.low + 1). For bounds 10..20, size is 11, so the one-argument constructor makes bounds 0..10. It copies data values but loses the original low/high bounds.

Q13. What happens after deleting a property? 
p = Person(20)
del p.age
print(p.age)

Result: the deleter deletes p._age, then the final print raises AttributeError.

Reasoning: The class still has the age property. The getter still runs. But the backing instance attribute is missing.

Q14. What prints? Class method and subclass construction. 
class Rat:
    def __init__(self, n=0, d=1):
        self.n = n
        self.d = d

    @classmethod
    def from_string(cls, s):
        n, d = s.split("/")
        return cls(int(n), int(d))

class LoudRat(Rat):

Output:

Reasoning: cls is the class the method was called on. Since the call is LoudRat.from_string(...), cls is LoudRat, not necessarily Rat.

Likely Questions Not Explicitly Shown but Naturally Testable

1. Why should unsupported arithmetic return NotImplemented instead of nothing?

If __add__ returns NotImplemented, Python can try the other operand's reflected method. If the method falls off the end and returns None, the expression can silently produce wrong behavior or fail with a confusing error later. The PDF's operator examples are about dispatch, so a hard exam may test the fallback path.

2. What is the difference between __str__ and __repr__?

The PDF uses __str__ as the replacement for C++ << printing. str(obj) and print(obj) call __str__. repr(obj), interactive display, and containers often use __repr__. If only __str__ is defined, printing the object itself looks nice, but a list of Rats may still show default object representations unless __repr__ is also defined.

3. Why does SA need both __getitem__ and __setitem__?

sa[i] calls __getitem__. sa[i] = value calls __setitem__. C++ can return an int& from operator[] so the same operator supports reading and writing. Python separates the read and write hooks.

4. If Python has garbage collection, why might a deleter still matter?

The PDF's deleter section is not about destroying the whole object. It is about controlled deletion of a property. A deleter can close a resource, log the deletion, maintain internal state, or reject deletion based on business rules. Garbage collection only handles object cleanup when objects become unreachable.

5. What is name mangling, exactly?

Inside class MyClass, an attribute named __x is internally stored as something like _MyClass__x. This prevents accidental collision in subclasses. It does not make the attribute impossible to access.

6. Why is [0] * n okay in the slide's SA, but sometimes dangerous?

It is okay here because 0 is immutable. Every slot can refer to the same integer object without mutation problems. It is dangerous with nested mutable objects, like [[]] * 3, because all slots reference the same inner list.

7. What is the difference between Rat.count += 1 and self.count += 1?

Rat.count += 1 definitely updates the class variable on Rat. self.count += 1 first reads through normal lookup, then assigns back to the instance, often creating or updating an instance variable that shadows the class variable. For counters shared by all objects, use the class name or a class method with cls.

8. Does del obj.my_property delete the property from the class?

No. It invokes the property deleter for that instance. The property object remains in the class definition. Other instances still have the property behavior.

Common Traps for This PDF

  • Fractions The shown Rat class does not simplify fractions. Rat(3, 4) * Rat(2, 5) prints 6/20, not 3/10.
  • Right-hand ops 3 + r is not r.__add__(3) first. Python tries the left operand first, then the right operand's reflected method.
  • Slide bug The page 3 __rsub__ needs __neg__ but does not define it. Exact slide code can fail for int - Rat.
  • One constructor Python does not choose between multiple __init__ definitions. Later definitions replace earlier ones.
  • self self is explicit in the method definition but automatically supplied in normal instance method calls.
  • cls In a class method, cls is the class used for the call. It can be a subclass.
  • Privacy _x is convention. __x is name-mangled. Neither is absolute privacy like C++ private.
  • Properties self.d = d can call a setter. It is not always a simple dictionary assignment.
  • Deleters A property deleter changes one instance's state. It does not remove the property from the class.
  • Class variables Assigning obj.count = 10 usually creates an instance variable; it does not necessarily update the class variable.
  • Static methods A static method receives no automatic self or cls. If it uses Rat.count, it is hard-coded to Rat.
  • SA indexing In SA(10, 20), external index 10 maps to internal list index 0.
  • Slide copy issue The page 17 __copy__ copies data but not original bounds for nonzero-low arrays.
  • Assignment b = a never calls __copy__. It only binds another name to the same object.
