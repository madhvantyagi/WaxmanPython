One-Page Exam Map

Pages 1-14 The lecture builds one idea in layers: Python can process a stream of values without storing the whole stream, because iteration is a protocol. Generators are a convenient way to write your own iterators while preserving local state between values.

Concept What the exam expects Key behavior
sum([1,2,3,4,5]) Recognize that the list is already fully built. All five integers exist in the list before sum starts.
sum(range(1,6)) Recognize that range is lazy and compact. The range object describes start, stop, step; it does not store a list of all values.
sum(i for i in range(1,6)) Recognize a generator expression. Values are produced one at a time and immediately consumed by sum.
Iterable Know it can produce an iterator through iter(obj) or obj.__iter__(). Examples: list, tuple, set, dictionary, string, file object, range, generator object.
Iterator Know it has __next__() and returns itself from __iter__(). Once exhausted, further next calls raise StopIteration.
Generator function Identify it by the presence of yield in a def. Calling it returns a generator object; the function body does not run yet.
yield Explain that it returns one value and suspends the function. Local variables, instruction pointer, and execution environment are retained.
list(generator) Explain how a constructor consumes an iterator. Repeatedly calls next, stores yielded values, stops when StopIteration occurs.
Coroutine style Know why next(monitor) is needed before send(value). The first next advances to the first yield; later send injects data into the suspended yield expression.
Restartable iteration Know why a generator object is one-use and why a class can restart. A class with __iter__ as a generator creates a fresh generator each time.

List vs Range vs Generator Expression

Pages 1-2 The opening question asks why the following all print 15 but are not the same:

print(sum([1, 2, 3, 4, 5]))
print(sum(range(1, 6)))
print(sum(i for i in range(1, 6)))

They have the same final value because each supplies sum with the numbers 1 through 5. They differ in how those numbers are represented before and during iteration.

List version

The list literal stores all elements at once. This is clear and direct, but for a huge input it means memory grows with the number of elements.

range version

range(1, 6) is a compact iterable. It represents the sequence without building a full list. During iteration, values are produced as needed.

Generator expression version

(i for i in range(1, 6)) creates a generator object. It is especially useful when each element needs a condition or transformation before being consumed.

Exam trap

Same printed output does not mean same memory behavior. A hard question may ask for the output and the memory model.

Behind the scenes in sum

sum does not need to know whether it was given a list, a range, or a generator expression. It only needs an iterable. Internally, the logic is conceptually like this:

total = 0
iterator = iter(input_object)
while True:
    try:
        item = next(iterator)
    except StopIteration:
        break
    total += item

That pattern is the central hidden mechanism behind most of the lecture.

The "Next Even Number" Problem

Page 2 The lecture asks for a function f that returns the next positive even integer less than 10 each time it is called: 2, then 4, then 6, then 8. Then it asks to do it without global variables.

This problem is really about state. A normal function starts fresh on every call unless the state is stored somewhere. The lecture later shows that generators solve this naturally.

Generator version

def even_less_than_10():
    n = 2
    while n < 10:
        yield n
        n += 2

f = even_less_than_10()
print(next(f))  # 2
print(next(f))  # 4
print(next(f))  # 6
print(next(f))  # 8

The variable n is local, not global. It survives between calls because the generator is suspended at yield, not destroyed.

What happens after 8?

print(next(f))

The loop condition n < 10 is false, the function reaches its end, and Python raises StopIteration. That is not a printed value; it is an exception.

Iterable and Iterator Protocol

Pages 2-3 The lecture reviews two related terms that are easy to blur together.

Term Definition Required methods Example
Iterable An object that can return its members one at a time. __iter__(), which returns an iterator. [1, 2, 3], "abc", range(5), file objects.
Iterator An object that represents a stream of data. __iter__() returning itself and __next__() returning the next item. iter([1, 2, 3]), generator objects.

Correct mental model

1. Iterablemy_list = [1, 2, 3]
2. Get iteratorit = iter(my_list)
3. Ask nextnext(it) returns 1
4. Ask againreturns 2, then 3
5. Endnext call raises StopIteration

Equivalent calls

my_list = [1, 2, 3]
my_iterator = iter(my_list)

print(next(my_iterator))        # 1
print(my_iterator.__next__())   # 2
print(next(my_iterator))        # 3
print(next(my_iterator))        # StopIteration

Important source-specific formatting trap

Page 3 lists examples that can look like separate snippets, not one continuous program. If you run this as one program:

my_list = [1, 2, 3]
my_iterator = iter(my_list)

for x in my_iterator:
    print(x)

print(next(my_iterator))

The final next(my_iterator) raises StopIteration because the for loop already consumed the same iterator. For output questions, always ask whether a fresh iterator was created.

Generator Functions and yield

Pages 3-5 A generator function is a function written with def that contains yield. Calling it does not run the function body. Calling it returns a generator object.

Even numbers example

def even_numbers(n):
    i = 0
    while i < n:
        yield i
        i += 2

even_gen = even_numbers(10)
for num in even_gen:
    print(num)

Output:

0
2
4
6
8

What yield actually does

When next(even_gen) is called, Python resumes the generator body. It runs until the next yield. At yield i, Python returns the current value of i to the caller and freezes the generator frame. The local variable i, the loop condition, and the exact next instruction are kept alive.

When next(even_gen) is called again, execution resumes immediately after the yield, so i += 2 runs before the loop checks again.

Countdown example

Pages 4-5 The lecture's countdown generator is the best output-prediction example because it contains a print before the loop.

def countdown(n):
    print('Counting down from', n)
    while n > 0:
        yield n
        n -= 1

c = countdown(10)

At this point, nothing from inside countdown has printed yet. The function has not begun executing. Python has only created a generator object.

print(next(c))

Output:

Counting down from 10
10

Reason: the first next starts the body, prints the message, enters the loop, yields 10, and then print prints that yielded value.

print(next(c))
print(c.__next__())

Output after the first call above has already happened:

9
8

next(c) and c.__next__() both advance the same iterator.

Generator object identity

Page 5 In the interpreter, evaluating c shows something like:

<generator object countdown at 0x105f73740>

The exact memory address changes every run. The exam should not expect the same address. It matters only that c is a generator object, not the first yielded number.

StopIteration, Return, and Exhaustion

Pages 2, 10, 13 An iterator signals that it is finished by raising StopIteration. Generator functions do this automatically when they reach the end of the function. They also do it when they execute a return statement.

def g():
    yield "A"
    yield "B"

x = g()
print(next(x))
print(next(x))
print(next(x))

Output and then exception:

A
B
StopIteration

Why for loops do not show the exception

A for loop catches StopIteration internally to know when to stop. Calling next manually does not hide it.

Return in a generator

def h():
    yield 1
    return
    yield 2

for item in h():
    print(item)

Output:

1

The return ends the generator. The later yield 2 is never reached.

How list() Consumes a Generator

Pages 8-10 The lecture asks why the list constructor performs iteration. The answer is that list(iterable) constructs a list by consuming the iterable one item at a time.

output = list(monitor)

Conceptually, this does:

output = []
it = iter(monitor)
while True:
    try:
        output.append(next(it))
    except StopIteration:
        break

The final list is no longer lazy. The generator was lazy while producing values, but list stores every yielded value in memory.

Memory trap

A generator saves memory only while you stream from it. If you immediately do list(generator), you ask Python to store all generated values.

Source Problem Patterns and Model Solutions

The PDF contains several generator-writing prompts. These are likely exam templates because they test state retention, lazy output, and correct stopping behavior.

1. Next prime number generator

Page 5 Each next call should return the next prime.

def gen_primes():
    candidate = 2
    while True:
        is_prime = True
        divisor = 2
        while divisor * divisor <= candidate:
            if candidate % divisor == 0:
                is_prime = False
                break
            divisor += 1
        if is_prime:
            yield candidate
        candidate += 1

State retained between calls: candidate. The generator does not recompute earlier primes unless you create a new generator.

2. Next leap year generator

Page 6 Leap year rules are naturally testable even if the PDF does not spell them out: divisible by 4, except centuries must be divisible by 400.

def gen_leap_years(start_year=1):
    year = start_year
    while True:
        if year % 400 == 0 or (year % 4 == 0 and year % 100 != 0):
            yield year
        year += 1

Example: 1900 is not a leap year, but 2000 is.

3. Dictionary mapping i to the ith prime

Page 6 The lecture gives the intended structure:

def prime_dict(n):
    prime_generator = gen_primes()
    return {i: next(prime_generator) for i in range(1, n + 1)}

For prime_dict(5), the result is:

{1: 2, 2: 3, 3: 5, 4: 7, 5: 11}

This is a dictionary comprehension. It consumes the same generator object repeatedly, so the prime sequence advances.

4. Octal number generator returning strings

Page 7 The source states that when decimal 16 is produced, the generator returns the string '20', because 20 is octal for decimal 16.

def get_oct():
    n = 0
    while True:
        yield format(n, "o")
        n += 1

First values: '0', '1', '2', ..., '7', '10', '11'. If the professor wants positive octal numbers only, start n = 1.

5. Fibonacci generator

Page 7 A Fibonacci generator is a classic state-retention test because it needs two previous values.

def fib():
    a = 0
    b = 1
    while True:
        yield a
        a, b = b, a + b

First values: 0, 1, 1, 2, 3, 5, 8, 13. If the class convention starts Fibonacci at 1, 1, initialize a = 1 and b = 1.

6. Serial number generator

Page 7 Serial numbers are ten-character strings. The first two characters are uppercase alphabetic from AA through ZZ in lexicographic order. The following eight characters are numeric from 00000000 through 99999999.

def serial_numbers():
    for first in range(ord("A"), ord("Z") + 1):
        for second in range(ord("A"), ord("Z") + 1):
            prefix = chr(first) + chr(second)
            for number in range(100000000):
                yield prefix + f"{number:08d}"

First values:

AA00000000
AA00000001
AA00000002

After AA99999999, the next value is AB00000000.

Coroutines and send

Pages 10-12 The lecture then says the list-based sensor example is unrealistic because the generator gets all inputs at once. In a more realistic setup, a producer sends one state at a time and a consumer keeps the gap state alive.

def sensor_monitor():
    zero_count = 0
    while True:
        state = (yield zero_count)
        if state == 1:
            zero_count = 0
        else:
            zero_count += 1

Why priming is required

monitor = sensor_monitor()
next(monitor)           # prime the generator
gap = monitor.send(0)   # now a value can be received by yield

The first next(monitor) runs the generator until it reaches yield zero_count. At that moment it yields 0 and pauses. The generator is now suspended exactly at the yield expression, ready for send to provide the value assigned to state.

Hard output trap

In state = (yield zero_count), the value yielded to the caller is the old zero_count. The value sent back in by the caller becomes state when the generator resumes.

Manual trace

m = sensor_monitor()
print(next(m))
print(m.send(0))
print(m.send(0))
print(m.send(1))
print(m.send(0))

Output:

0
1
2
0
1

Reason:

  1. next(m) reaches yield zero_count and yields 0.
  2. send(0) sets state = 0, increments zero_count to 1, loops, yields 1.
  3. send(0) increments to 2, yields 2.
  4. send(1) resets to 0, yields 0.
  5. send(0) increments to 1, yields 1.

Producer-consumer simulation

Pages 11-12 The lecture uses random.randint(0, 1) to simulate a producer and monitor.send(produced) to simulate a consumer processing that value.

import random

def main():
    monitor = sensor_monitor()
    next(monitor)
    for _ in range(20):
        produced = random.randint(0, 1)
        print(f"Produced: {produced}")
        gap = monitor.send(produced)
        print(f"Consumed: {produced}, Gap: {gap}")

The exact output changes because random values change. The rule does not change: each sent 0 increases the gap; each sent 1 resets it.

Security sensor warning version

Page 12 The source asks for a modified generator: if the gap between available signals exceeds a user-defined n, return a warning and return.

def security_monitor(limit):
    zero_count = 0
    while True:
        state = (yield zero_count)
        if state == 1:
            zero_count = 0
        else:
            zero_count += 1
            if zero_count > limit:
                yield "WARNING"
                return

Trace:

m = security_monitor(2)
print(next(m))      # 0
print(m.send(0))    # 1
print(m.send(0))    # 2
print(m.send(0))    # WARNING
print(m.send(1))    # StopIteration

After return, the coroutine is exhausted. It cannot be reused.

One-Use Generator Objects vs Restartable Iterables

Pages 13-14 A generator object executes once. After it is exhausted, iterating over it again produces nothing.

c = countdown(3)
for n in c:
    print('T-minus', n)

for n in c:
    print('T-minus', n)

Output:

Counting down from 3
T-minus 3
T-minus 2
T-minus 1

The second loop prints nothing because c is already exhausted.

Restartable class pattern

class countdown:
    def __init__(self, start):
        self.start = start

    def __iter__(self):
        n = self.start
        while n > 0:
            yield n
            n -= 1

Now each loop calls __iter__, and __iter__ creates a fresh generator.

c = countdown(3)
for n in c:
    print(n)
for n in c:
    print(n)

Output:

3
2
1
3
2
1

Common Traps From This Lecture

Calling a generator function runs the body

False. c = countdown(10) creates a generator object. The body starts on next(c), a for loop, list(c), sum(c), or another consumer.

A generator expression stores all values

False. It produces one value at a time. But list(generator_expression) stores them all.

return yields a value

False for normal iteration. A return ends the generator and causes StopIteration.

A generator object restarts automatically

False. A generator object is one-use. Use a function call again or a class whose __iter__ creates a fresh generator.

send can send a non-None value into a new generator

False. You must prime the generator first, usually with next(g). A just-created generator has not reached a yield expression yet.

Every input to sensor_monitor(states) causes an output

False. In the list-based version, only a 1 causes a yield. Zeros only increase the internal gap.

Professor-Style Output and Reasoning Questions

Use these like exam drills. Predict first, then open the answer.

Question 1: Generator creation vs execution

def countdown(n):
    print('Counting down from', n)
    while n > 0:
        yield n
        n -= 1

c = countdown(3)
print("A")
print(next(c))
print("B")
Answer and reasoning 
A
Counting down from 3
3
B

c = countdown(3) does not run the body. print("A") runs first. Then next(c) starts the generator, so it prints Counting down from 3, yields 3, and the outer print prints 3.

Question 2: Where does resume happen?

def g():
    x = 10
    yield x
    x += 5
    yield x
    x += 5
    yield x

a = g()
print(next(a))
print(next(a))
print(next(a))
Answer and reasoning 
10
15
20

After the first yield x, the generator pauses before x += 5. The second next resumes after the first yield, updates x to 15, and yields. The third repeats that pattern.

Question 3: Exhaustion

def g():
    yield 1
    yield 2

x = g()
print(list(x))
print(list(x))
Answer and reasoning 
[1, 2]
[]

The first list(x) consumes the generator until StopIteration. The same generator object is exhausted, so the second list(x) finds no values.

Question 4: Fresh generator each time

def g():
    yield 1
    yield 2

print(list(g()))
print(list(g()))
Answer and reasoning 
[1, 2]
[1, 2]

Each call to g() creates a new generator object, so both lists get the full sequence.

Question 5: Iterator consumed by a loop

nums = [1, 2, 3]
it = iter(nums)
for x in it:
    print(x)
print(next(it))
Answer and reasoning 
1
2
3
StopIteration

The first three lines are printed by the loop. The loop stops because the iterator is exhausted. The explicit next(it) then raises StopIteration. In a real traceback, StopIteration appears as an exception, not a normal printed line.

Question 6: List is restartable, iterator is not

nums = [1, 2, 3]
for x in nums:
    print(x)
for x in nums:
    print(x)
Answer and reasoning 
1

3

3

A list is an iterable that creates a fresh iterator for each loop. This is different from reusing the same iterator object.

Question 7: Sensor monitor output

def sensor_monitor(states):
    gap = 0
    for state in states:
        if state == 1:
            yield gap
            gap = 0
        else:
            gap += 1

print(list(sensor_monitor([0, 0, 1, 0, 1, 1, 0])))
Answer and reasoning 
[2, 1, 0]

The first two zeros build gap 2. The first 1 yields 2 and resets. Then one zero builds gap 1, the next 1 yields 1, and the next 1 yields 0. The final zero does not yield because no later 1 appears.

Question 8: list() triggers execution

def g():
    print("start")
    yield 1
    print("middle")
    yield 2
    print("end")

print("before")
print(list(g()))
print("after")
Answer and reasoning 
before
start
middle
end
[1, 2]
after

The outer print(list(g())) must finish constructing the list before it can print the list. During construction, list repeatedly calls next, so all internal prints happen first. The end print happens after the second yield resumes and before the generator reaches the end.

Question 9: sum with a generator expression

def noisy(n):
    print("make", n)
    return n

print(sum(noisy(i) for i in range(3)))
Answer and reasoning 
make 0
make 1
make 2
3

The generator expression calls noisy(i) lazily as sum asks for each item. The final sum is 0 + 1 + 2.

Question 10: yield before update

def even_numbers(n):
    i = 0
    while i < n:
        yield i
        i += 2

e = even_numbers(5)
print(next(e))

Answer and reasoning

2

StopIteration

The yielded values are 0, 2, and 4. After yielding 4, the next resume runs i += 2, making i = 6. The loop condition fails and the generator ends.

Question 11: Coroutine priming mistake

def sensor_monitor():
    zero_count = 0
    while True:
        state = (yield zero_count)
        if state == 1:
            zero_count = 0
        else:
            zero_count += 1

m = sensor_monitor()
print(m.send(0))
Answer and reasoning 
TypeError

You cannot send a non-None value into a just-created generator. The generator has not yet reached a yield expression where the value can be received. Prime it first with next(m) or m.send(None).

Question 12: Coroutine send trace

0

Priming yields 0. Sending 1 resets to 0 and yields 0. The next two zeros increase the gap to 1 and 2. Sending 1 resets it again.

Question 15: Restartable class

class countdown:
    def __init__(self, start):
        self.start = start

    def __iter__(self):
        n = self.start
        while n > 0:
            yield n
            n -= 1

print(list(c))
print(list(c))
Answer and reasoning 
[2, 1]
[2, 1]

Each list(c) calls c.__iter__(), and each call creates a fresh generator from the beginning.

Question 16: Generator object inside iter

g = even_numbers(4)
print(g is iter(g))
print(next(g))

Answer and reasoning 
True
0
2

A generator object is its own iterator, so iter(g) returns g itself. It does not create a fresh generator.

Question 17: Likely exam extension - filter and transform

print(sum(i * i for i in range(6) if i % 2 == 0))
Answer and reasoning 
20

The even values from range(6) are 0, 2, and 4. Squared, they are 0, 4, and 16. The sum is 20. This tests the page 1 point that generator expressions are flexible for transformations and conditions.

Question 18: Likely exam extension - hidden memory issue

values = (i for i in range(1000000000))
first_three = [next(values), next(values), next(values)]
print(first_three)
Answer and reasoning 
[0, 1, 2]

The generator expression does not build a billion integers. It keeps enough state to know the next value from the range iterator. Only three values are requested.

How to Answer Written Exam Questions

When a problem asks you to explain a generator, use this checklist:

  1. State whether the object is an iterable, an iterator, or both.
  2. Identify when execution begins. For generator functions, it begins on first consumption, not on function call.
  3. Identify where execution pauses. It pauses at yield.
  4. Identify which local variables survive. Those are the generator's retained state.
  5. Identify who is consuming the generator: for, next, list, sum, dictionary comprehension, or send.
  6. Identify how it ends: loop finishes, function reaches the end, or return executes, causing StopIteration.
  7. For memory questions, separate lazy generation from forced collection. list(g) is not memory-light after it completes.
